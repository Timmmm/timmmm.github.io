Using std in embedded Rust

19th April 2021

I recently asked a question about using the Rust std library in microcontrollers. It was closed by the StackOverflow question Nazis (quelle surprise), but I have figured out the answer myself so I will post it here where they can‚Äôt unilaterally execute it.

The question is: Can you use std in embedded Rust programming?

A lot of readers are probably instinctively thinking ‚Äúyou shouldn‚Äôt do that!‚Äù, often for reasons like this:

on a microcontroller you generally don‚Äôt have enough memory to rely on a heap just being big enough for things like Vecs, so you just use arrays.

That‚Äôs nonsense. Modern microcontrollers can have 1 MB of RAM or even more. That‚Äôs the minimum requirement for Windows 3.0! Also it‚Äôs common to use heap-allocated objects that are only allocated once at startup.

The real reason not to use a heap on microcontrollers is because it can fail at runtime, which on safety critical or high availability devices is a problem. But that doesn‚Äôt apply in most applications. Mbed and Arduino have had heaps for years.

My original motivation

My original motivation was to be able to use containers like Vec and VecDeque. For some simple containers there is the heapless crate that provides non-heap versions of containers, but it doesn‚Äôt cover everything - for example it doesn‚Äôt include a version of VecDeque.

One of the StackOverflow comments pointed me to an easier solution: you can use the alloc crate (not to be confused with core::alloc) even in #![no_std] programs. alloc contains most of the functionality from std you might want to use on bare metal that isn‚Äôt in core. You just have to provide a memory allocator. A simple allocator is linked-list-allocator. Let‚Äôs try it out!

Trying out alloc

I followed the excellent Rust Embedded Book‚Äôs guide to running a Cortex-M3 program on QEMU. Following the guide we get to the point where we can cross-compile the following example code to thumbv7m-none-eabi and run it.

#![no_std]

use panic_halt as _;

use cortex_m_rt::entry;
use cortex_m_semihosting::{debug, hprintln};

    hprintln!("Hello, world!").unwrap();

    debug::exit(debug::EXIT_SUCCESS);

}

Ok there are two problems here:

  1. alloc is unstable so we have to use nightly, and
  2. we haven‚Äôt provided an allocator.

To add the allocator we can uncomment the cortex-m-alloc dependency in Cargo.toml:

# Uncomment for the allocator example.
alloc-cortex-m = "0.4.0"

alloc-cortex-m is a wrapper around linked-list-allocator that makes it interrupt safe for Cortex-M processors.

Then we can use it in the code. Conveniently there‚Äôs an example at examples/allocator.rs that we can copy & paste from:

#![no_main]
#![no_std]
#![feature(alloc_error_handler)] // üëà

use panic_halt as _;

use cortex_m_rt::entry;
use cortex_m_semihosting::{debug, hprintln};

extern crate alloc;
use alloc::vec::Vec;

use alloc_cortex_m::CortexMHeap;                        // üëà
                                                        // üëà
// this is the allocator the application will use       // üëà
#[global_allocator]                                     // üëà
static ALLOCATOR: CortexMHeap = CortexMHeap::empty();   // üëà
                                                        // üëà
const HEAP_SIZE: usize = 1024; // in bytes              // üëà

#[entry]
fn main() -> ! {
    // Initialize the allocator BEFORE you use it                             // üëà
    unsafe { ALLOCATOR.init(cortex_m_rt::heap_start() as usize, HEAP_SIZE) }  // üëà

    let mut v = Vec::new();
    v.push(5);
    v.push(2);
    v.push(5);
    for x in v.iter() {
      hprintln!("Hello {}", x).unwrap();
    }

    debug::exit(debug::EXIT_SUCCESS);
    loop {}
}

// define what happens in an Out Of Memory (OOM) condition     // üëà
#[alloc_error_handler]                                         // üëà
fn alloc_error(_layout: core::alloc::Layout) -> ! {            // üëà
    loop {}                                                    // üëà
}                                                              // üëà

What about full std?

So this still left me wondering, what if I really do want to have std available? Can you do it? Let‚Äôs try!

What determines if std is available? It is sort of dictated by the target you are compiling for. The target is described by a ‚Äútarget triple‚Äù which is a barely consistent tuple of up to four parts (yep it‚Äôs inconsistent and badly named) describing the target architecture and platform that you are compiling for. There is a list of supported targets here.

Notice that some of them do not support std (indicated by a *). For example thumbv7m-none-eabi which we are using does not, but wasm32-unknown-unknown does. Rust sensibly does not try to parse the target triple. Instead each target has a target spec, which you can print like this:

Note the line "os": "unknown",. According to this comment:

    /// "none" implies a bare metal target without `std` library.
    /// A couple of targets having `std` also use "unknown" as an `os` value,

Roughly, targets with "os": "none" (which is the default) do not have std available. However that‚Äôs not exactly how it works. You can actually build std for any target, even targets with "os": "none". Let‚Äôs see how.

There‚Äôs not much documentation on this error, but I managed to figure out what this means.

Basically platform-dependent parts of std - such as threading and filesystem access - are in the sys crate. There is an implementation of sys inside std for each platform that Rust supports. The selected implementation is determined mostly by the os field of the target spec. If the os is none then it uses the unsupported implementation in which all operations fail or do nothing.

When std is built, if the target does not match one of these whitelisted targets then the restricted_std feature is required to indicate that that build of std is not supported. I am not really sure why the whitelist does not match the logic for selecting the sys implementation but I‚Äôm sure there‚Äôs a reason.

To tell Rust we‚Äôre ok with using the unsupported std, just add #![feature(restricted_std)] at the start of main.rs.

Why this isn‚Äôt stupid

In most bare metal situations you only need core and alloc. std mostly exposes items from core or alloc. The exceptions are functions normally provided by an OS like network access, running processes and using files. Those things rely on the sys crate.

Those operations do make sense on bare metal! We might want to provide a virtual filesystem for example, and plenty of microcontrollers support network features. Unfortunately the implementations are all embedded in the compiler - there is no way to define your own though that is planned eventually:

//! In the future it would be desirable for the independent
//! implementations of this module to be extracted to their own crates
//! that `std` can link to, thus enabling their implementation
//! out-of-tree via crate replacement. Though due to the complex
//! inter-dependencies within `std` that will be a challenging goal to
//! achieve.

However there are still some compelling reasons to use std:

  1. HashMap is only available in std.
  2. Error is only available in std.
  3. You can use std crates.
  4. async/await only works in std Edit: maybe not.

Unfortunately there are some serious caveats so I would probably stick to core and alloc for now. Specifically:

  1. You can‚Äôt use HashMap with the default hasher because it uses thread_local!() which requires support from sys.
  2. I think the same issue applies to async/await though I haven‚Äôt tested it.
  3. You actually can‚Äôt use std dependencies because there‚Äôs no way to pass #[feature(restricted_std)] to them as far as I can tell. Every crate needs that feature, not just the root one.

Anyway here is a simple program to demonstrate some std-only features.

#![no_main]
#![feature(alloc_error_handler)]
#![feature(restricted_std)]

use cortex_m_rt::entry;
use cortex_m_semihosting::{debug, hprintln};

use std::{
    collections::{HashMap, hash_map::DefaultHasher},
    hash::{BuildHasher, BuildHasherDefault},
};

                                                         // üëá
type Result<T> = std::result::Result<T, Box<dyn std::error::Error>>;

use alloc_cortex_m::CortexMHeap;

// this is the allocator the application will use
#[global_allocator]
static ALLOCATOR: CortexMHeap = CortexMHeap::empty();

const HEAP_SIZE: usize = 1024; // in bytes

#[entry]
fn main() -> ! {
    // Initialize the allocator BEFORE you use it
    unsafe { ALLOCATOR.init(cortex_m_rt::heap_start() as usize, HEAP_SIZE) }

             // üëá
    let mut v = HashMap::with_hasher(BuildHasherDefault::<DefaultHasher>::default());
    v.insert(1, 1);
    match get_zero(&v) {
        Ok(x) => hprintln!("Element 0 is {}", x).unwrap(),
        Err(e) => hprintln!("Error: {}", e).unwrap(),
    }

    debug::exit(debug::EXIT_SUCCESS);
    loop {}
}

fn get_zero<S: BuildHasher>(x: &HashMap<u32, u32, S>) -> Result<u32> {
    Ok(*x.get(&0).ok_or("Missing element 0")?)
}
